# Task File Spec

Each task folder under `workspace/TASKS/<task_name>/` may contain:
- `task.md`: execution instructions for the agent
- `task.json`: structured task metadata for the task dashboard / UI

The goal of `task.md` is to define:
- what the agent is trying to achieve
- what tools or skills it must use
- what constraints it must respect
- what output it must return
- when it should stop and ask the user for help or approval

The goal of `task.json` is to provide:
- a machine-readable task record for the dashboard
- current task status and priority
- approvals, constraints, artifacts, and timeline entries
- a default chat/history state for the task UI

## Required Sections

Every `task.md` should use these sections in this order:

### 1. Title
Use a single H1 heading.

Example:

```md
# Task: Summarize Slack Starred Chats
```

### 2. Objective
Briefly describe the user-facing goal.

This should answer:
- what the agent is supposed to do
- what a successful result looks like

### 3. Context
Provide task-specific background the agent should know before acting.

This may include:
- product or business context
- important assumptions
- current state
- location of relevant files or services

### 4. Required Skills / Tools
Explicitly name the skills or tools the agent should use.

Rules:
- use exact repo skill names when possible
- distinguish between required and optional
- mention whether the task should use browser automation, APIs, local files, or shell tools

### 5. Workflow
Describe the expected execution steps.

Rules:
- keep steps concrete and action-oriented
- include login / auth handling if relevant
- include data collection and verification steps
- include how the agent should react if something cannot be accessed

### 6. Output Format
Define the final deliverable structure.

Rules:
- specify section names the agent must return
- specify whether the output should be short, medium, or detailed
- specify whether TODOs, risks, tables, or bullet lists are required

### 7. Constraints
List hard rules the agent must not violate.

Typical examples:
- do not send messages
- do not modify data
- do not take destructive actions
- do not guess when evidence is missing

### 8. Stop Conditions / Escalation
Define when the agent should stop and ask the user.

Examples:
- login required
- approval required
- required data missing
- ambiguous choice with irreversible consequences

## Recommended Optional Sections

Use these when helpful:

### Inputs
- files, folders, URLs, credentials, or IDs the agent expects

### Success Criteria
- concrete pass/fail conditions

### Deliverables
- files, summaries, screenshots, or reports the agent should produce

### Notes
- task-specific exceptions or edge cases

## Writing Guidelines

- Prefer direct instructions over vague goals.
- Write for execution, not discussion.
- Assume the agent should act unless a stop condition is hit.
- Be explicit about skills, permissions, and boundaries.
- If the task depends on external systems, say what to do when access fails.
- If the task involves summarization, tell the agent what to prioritize and what to ignore.

## Minimal Template

```md
# Task: <name>

## Objective
<what the agent should accomplish>

## Context
<background, assumptions, relevant state>

## Required Skills / Tools
- `<skill_or_tool_name>`

## Workflow
1. <step one>
2. <step two>
3. <step three>

## Output Format
### 1. <section name>
- <required content>

### 2. <section name>
- <required content>

## Constraints
- <rule>
- <rule>

## Stop Conditions / Escalation
- <when to stop and ask the user>
```

## TASK.json Spec

`task.json` should contain either:
- a single task object, or
- an object with a top-level `tasks` array

The dashboard currently expects task records compatible with the examples in [`example_tasks.json`](/Users/liqiangx/Documents/code/EasyClaw/workspace/TASKS/example_tasks.json).

### Recommended Top-Level Shape

```json
  {
    "id": "task-id",
    "title": "Task title",
    "status": "Planning",
    "priority": "Medium",
    "risk_level": "Low",
    "goal": "What success looks like",
    "period": "every weekday 5:00pm",
    "last_execution_timestamp": "2026-04-01T17:00:00-07:00",
    "next_execution_timestamp": "2026-04-02T17:00:00-07:00",
    "next_action": "Next step",
    "current_step": "Current state",
    "approvals": [],
    "constraints": [
      "Do not do any risky things"
    ],
    "history": [
      {"execution_timestamp": "2026-03-31T17:00:00-07:00",
      "results": "...."
      },
      {"execution_timestamp": "2026-03-31T17:00:00-07:00",
      "results": "...."
      }
    ]
  }
```

## Required Fields For Each Task Record

- `id`: stable unique identifier
- `title`: short user-facing title
- `status`: one of the dashboard-friendly states such as `Draft`, `Planning`, `In progress`, `Waiting on me`, `Waiting on others`, `Blocked`, `Done`, or `Failed`
- `priority`: usually `Low`, `Medium`, or `High`
- `goal`: one-sentence task goal
- `period`: recurrence string such as `no repeat`, `every weekday 5:00pm`, or `every day morning 5am`
- `next_execution_timestamp`: next scheduled execution time in ISO 8601 format when applicable
- `history`

## Recommended Fields

- `last_update`
- `next_action`
- `current_step`
- `risk_level`
- `approvals`
- `constraints`
- `last_execution_timestamp`: most recent execution time in ISO 8601 format when applicable


## Nested Field Conventions

### timestamps

For machine-readable timestamps, use ISO 8601.

Recommended format:

```json
"next_execution_timestamp": "2026-04-01T17:00:00-07:00"
```

Rules:
- include timezone information whenever possible
- prefer full datetime over date-only values
- use `Z` for UTC if appropriate, for example `"2026-04-02T00:00:00Z"`
- do not use the misspelled field name `next_excuation_timestamp`

### period

Use a short stable recurrence string.

Examples:

```json
"period": "no repeat"
```

```json
"period": "every weekday 5:00pm"
```

```json
"period": "March 21 5:00pm"
```


Rules:
- use `no repeat` for one-time tasks
- keep the format consistent across tasks
- recurring tasks should usually include both `last_execution_timestamp` and `next_execution_timestamp`

### approvals

Each approval item should look like:

```json
{
  "id": "approval-id",
  "title": "Approval title",
  "reason": "Why approval is needed",
  "status": "pending"
}
```


## Writing Guidelines For task.json

- Keep values short and dashboard-friendly.
- Use `next_action` for the most important immediate step.
- Use `approvals` only for true user decisions or permissions.
- Use `history` to append results every execution, not every tiny event.
- Prefer one folder per task and one `task.json` per task family or task collection.
